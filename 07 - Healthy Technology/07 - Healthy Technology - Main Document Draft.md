# 07 - Healthy Technology
## Frankston 2035 - Art That Lives
### Main Document Draft

## Charter title

Healthy Technology Charter: Technology Should Give Attention Back

## Charter status

Draft for Council, school, library, family, artist, and community review.

This is a working standard for public-facing pilots. It should be reviewed before any child-facing, school-facing, library-facing, QR-linked, or public-art-linked digital experience is launched.

---

## 1. Purpose

Frankston 2035 - Art That Lives may use digital tools to support public art, learning, walking trails, workshops, poster projects, school resources, family prompts, and community participation.

This charter sets the safety and design rules for those tools.

The aim is simple:

Technology should give attention back.

Digital layers should help people notice Frankston, make something, learn something, share something useful, or participate in a local creative act. They should not pull people into endless screen use.

---

## 2. Definition of healthy technology

Healthy technology is technology that:

- serves a clear civic, educational, creative, or access purpose
- works without a login wherever possible
- avoids collecting personal data, especially from children
- supports short sessions and clear endings
- leads to offline action, reflection, making, walking, discussion, or learning
- is understandable to a parent, teacher, librarian, or council officer
- avoids manipulative interface design
- can be used in a classroom, library, family, or public setting without creating avoidable risk

Healthy technology does not compete with Frankston for attention. It points attention back to Frankston.

---

## 3. Scope

This charter applies to:

- QR-linked public art pages
- school and classroom learning tools
- library workshop resources
- family creative prompts
- poster seed pack interactions
- community-made resource libraries
- walking trail web layers
- sister-city exchange pages
- any Blue Snake Studio digital tool used in the Frankston 2035 ecosystem

It applies most strongly when children, schools, families, public venues, or council-supported programs are involved.

---

## 4. Core principles

### 4.1 Frankston is the hero

The technology must make Frankston easier to notice, explore, learn from, or contribute to.

It must not make Blue Snake Studio the centre of the experience.

### 4.2 Create, then leave the screen

The preferred user journey is:

1. Open the prompt.
2. Understand the task.
3. Make, notice, walk, draw, discuss, or share offline.
4. Finish.

The screen is a doorway, not a destination.

### 4.3 Short sessions are a feature

A healthy session should usually take 30 seconds to 5 minutes unless the user is deliberately reading, downloading, or using a classroom resource.

Short sessions show respect for families, teachers, libraries, public places, and attention.

### 4.4 Clear endings

Every experience should make it obvious when the user is done.

Examples:

- "You have your prompt. Look up and find it."
- "Close the screen and draw for five minutes."
- "Take this back to class."
- "Print this for the workshop."
- "Walk to the next site."

### 4.5 No attention traps

The project rejects:

- infinite scroll
- autoplay feeds
- streaks
- loot boxes
- gambling-style rewards
- variable-ratio rewards
- disguised advertising
- guilt prompts
- forced notifications
- manipulative countdowns
- confusing opt-outs
- dark patterns

### 4.6 No child data by default

Child-facing tools must not collect personal data from children.

If a feature seems to require child data, the feature should be redesigned.

Acceptable alternatives may include:

- printed worksheets
- classroom group codes managed by adults
- no-save local activities
- anonymous aggregate counts
- teacher-mediated submissions
- opt-in adult-managed consent pathways

### 4.7 Anonymous aggregate metrics only where needed

Measurement should be minimal and proportionate.

Possible acceptable metrics:

- number of page visits
- number of poster downloads
- number of school/library sessions delivered
- number of printed packs distributed
- number of partner organisations involved
- broad location/site usage without identifying people

Metrics should not become the purpose of the project.

### 4.8 Accessibility is not optional

Every public-facing tool should meet basic accessibility expectations:

- readable contrast
- keyboard navigation
- mobile readability
- plain English
- alt text for meaningful images
- captions or transcripts where audio/video is used
- printable alternatives
- no essential meaning carried by colour alone
- simple page structure for assistive technology

### 4.9 Permission and cultural safety

Digital tools must not invent, simulate, decorate with, or extract First Nations language, symbols, stories, or cultural material.

Any First Nations content must be permission-based, respectful, and guided by appropriate Traditional Owner organisations.

Sister City content must focus on real exchange, learning, and relationship, not symbolic decoration.

## 5. Non-negotiable design standards

| Standard | Requirement | Reason |
|---|---|---|
| No infinite scroll | Pages must have clear boundaries and endings. | Prevents passive consumption. |
| No dark patterns | No manipulative consent, sign-up, navigation, or exit flows. | Protects trust. |
| No gambling mechanics | No loot boxes, streak pressure, random reward loops, or chance-based incentives. | Protects children and families. |
| No login by default | Public resources should work without accounts wherever possible. | Reduces exclusion and privacy risk. |
| No child personal data collection | Child-facing tools must not ask children for names, emails, photos, locations, profiles, or personal identifiers. | Reduces legal, ethical, and duty-of-care risk. |
| Short-session design | Most public interactions should be quick and purposeful. | Returns attention to place. |
| Clear endings | Every interaction should tell the user what to do next offline. | Supports healthy use and classroom control. |
| Offline creative prompts | Digital prompts should lead to walking, drawing, writing, making, observing, or discussion. | Makes technology useful beyond the screen. |
| Family/classroom modes | Tools should work for adult-led groups without individual child accounts. | Helps schools and families use the resource safely. |
| Accessibility basics | Public material must be readable, navigable, and printable. | Public art must be publicly usable. |
| Anonymous aggregate metrics only | Measure only what is needed to improve or report the pilot. | Avoids surveillance creep. |

## 6. Design rules

### Rule 1 - One purpose per tool

Each digital tool should do one clear job.

Example:

- "Give a walking prompt."
- "Open a mural lesson."
- "Download a poster."
- "Support a class discussion."

If a tool tries to be a social network, archive, game, classroom platform, analytics dashboard, and promotional campaign at once, it is too complex.

### Rule 2 - Explain it to a parent in one sentence

If the purpose cannot be explained plainly, simplify it.

Parent-safe sentence:

"This page gives your child a short creative prompt about Frankston, then asks them to make something offline."

### Rule 3 - Prefer no-login public access

No-login access should be the default for:

- poster pages
- walking prompts
- public art learning layers
- printable worksheets
- family prompts
- library activity pages

Logins should only be considered for adult-managed administration, not child participation.

### Rule 4 - Design for adults leading groups

Schools, libraries, and families need group-safe modes:

- class mode
- family mode
- workshop mode
- print mode
- low-bandwidth mode
- no-submission mode

The tool should help an adult run a session, not force every child into an individual account.

### Rule 5 - Build the exit

Every digital journey must include an intentional exit:

- "Close this and draw."
- "Walk to the next marker."
- "Talk to someone at your table."
- "Print this and use it offline."
- "Bring the idea to the next workshop."

### Rule 6 - Use metrics sparingly

Metrics should answer practical pilot questions:

- Did people find it?
- Did teachers use it?
- Did libraries distribute it?
- Did families understand it?
- Did it produce offline participation?

Metrics should not track individual children, profile households, or optimise for compulsive engagement.

### Rule 7 - Make the non-digital version strong

Every important public-facing digital tool should have a printable or offline equivalent where possible.

The project should still work if:

- a family has limited data
- a school blocks a site
- a library printer is the main access point
- a teacher wants a paper worksheet
- a public QR code is unavailable

### Rule 8 - Cut attention-stealing features

Reject features designed mainly to increase time on screen.

Examples to cut:

- endless recommendation feeds
- streak counters
- addictive reward loops
- push notifications for children
- artificial scarcity
- leaderboard pressure for children
- auto-playing content
- hidden exits

### Rule 9 - Design for public trust

Every public-facing page should make clear:

- who made it
- who it is for
- what it asks the user to do
- whether any data is measured
- what happens next

Blue Snake Studio can be present as a maker stamp or delivery note, but Frankston remains the hero.

## 7. Risk controls

| Risk | Control |
|---|---|
| Children provide personal data | Remove child data fields; use adult-mediated, paper, or anonymous pathways. |
| Tool becomes compulsive | Remove loops, streaks, endless feeds, and variable rewards. |
| Schools cannot approve use | Provide no-login, no-child-data, printable classroom mode. |
| Families distrust QR links | Use plain language, visible maker/council context, and clear privacy notes. |
| Library staff inherit support burden | Provide simple session cards, print packs, and a one-page facilitator note. |
| Public art becomes a gimmick | Require each QR layer to teach, prompt, or connect back to place. |
| Accessibility gaps exclude users | Check contrast, layout, alt text, keyboard use, print styles, and plain English. |
| Cultural content is mishandled | Do not create cultural content without permission and guidance. |
| Sister-city content becomes decorative | Use exchange formats: shared prompts, student work, artist notes, translated summaries where approved. |
| Metrics drift into surveillance | Use anonymous aggregate metrics only where needed. |

---

## 8. Pilot checklist

Before any public launch, answer these questions:

1. What is the one-sentence purpose?
2. Who is the intended user?
3. Does it work without login?
4. Does it collect any personal data?
5. Could a child use it without entering personal details?
6. Does it have a clear ending?
7. Does it lead to offline creativity, learning, noticing, walking, or discussion?
8. Is there a printable alternative?
9. Is the page accessible on mobile?
10. Can a teacher or librarian run it with a group?
11. Are privacy notes visible and plain?
12. Are any cultural permissions required?
13. Are any artist, copyright, site, or council permissions required?
14. What anonymous metrics are genuinely needed?
15. What would make this pilot stop?

---

## 9. What Council can safely support

Council can safely support:

- a no-login QR learning layer for one approved public art site
- printable creative prompts for libraries and schools
- a classroom-friendly mural lesson wall
- a family walking prompt with no data collection
- a community resource page with adult-submitted materials only
- a short workshop using printed worksheets and no child accounts
- a pilot evaluation based on anonymous aggregate usage and partner feedback

Council should be cautious about:

- tools that require children to create accounts
- features that ask children to upload photos, names, school details, or contact information
- public leaderboards involving children
- open comment systems
- unmoderated submissions
- AI-generated cultural content
- gamified reward mechanics
- any feature that cannot be explained to families plainly

---

## 10. What Blue Snake Studio must prove

Blue Snake Studio must prove that it can:

- build no-login public tools that are fast and readable
- avoid child data collection
- design short-session experiences with clear endings
- provide offline alternatives
- support teachers, libraries, families, and council officers with plain-language material
- use accessibility basics consistently
- handle cultural and sister-city material with restraint and permission
- measure only what is needed
- keep the public value visible and keep the studio brand secondary

---

## 11. What success looks like

Success looks like:

- a councillor understands the public value in under 60 seconds
- a teacher can use the resource without creating student accounts
- a library can print and run a session without special training
- a family can scan, understand, and leave the screen quickly
- a child makes something offline
- a public artwork becomes more useful without becoming a gimmick
- usage can be reported without identifying children
- Frankston feels more creative, not more digitised

---

## 12. What failure looks like

Failure looks like:

- users stay on screen longer without making or noticing anything
- a child is asked for personal information
- a feature needs an account before value is delivered
- teachers cannot use it because of privacy or classroom risk
- a QR code opens a confusing or promotional page
- families cannot tell what is being measured
- a feature imitates gambling or compulsive app patterns
- B$S branding overwhelms the civic purpose
- cultural material is used without permission or guidance
- the project looks innovative but does not help Frankston become more creative

---

## 13. Application to children

For children, healthy technology means:

- no personal data collection
- no accounts by default
- no open comments
- no public ranking pressure
- no gambling-style reward loops
- no hidden tracking
- prompts that lead to drawing, walking, building, discussing, or noticing
- adult-led modes for families, classes, and workshops
- clear endings that make it normal to stop

The design should assume that children deserve dignity, privacy, attention, and creative agency.

---

## 14. Application to schools

For schools, healthy technology means:

- teacher-led use
- printable worksheets
- class mode rather than individual accounts
- plain privacy position
- curriculum-adjacent prompts without pretending to replace curriculum
- low setup time
- no mandatory student uploads
- no behavioural hooks that make classroom management harder

School value should be practical:

- easy to trial
- safe to explain
- useful for one lesson
- expandable only after feedback

---

## 15. Application to libraries

For libraries, healthy technology means:

- public access without complex onboarding
- print packs
- facilitator notes
- drop-in workshop prompts
- simple QR pages
- accessibility-minded design
- no staff burden from account recovery, moderation, or technical support

Libraries should be able to host the work as a public learning resource, not troubleshoot a product.

---

## 16. Application to families

For families, healthy technology means:

- clear purpose
- no pressure to register
- no child data request
- short prompts
- low-cost or free activities
- outdoor, kitchen-table, library, or classroom use
- activities that help adults and children talk, notice, draw, build, or walk together

The design should support families who have limited time, limited data, mixed devices, or different confidence levels with technology.

---

## 17. Application to public art

For public art, healthy technology means:

- QR layers that add learning, access, or participation
- short explanations
- artist and permission respect
- offline prompts
- no gimmick overlays
- no app install requirement
- no attention trap beside an artwork

The digital layer should help the public return to the artwork, the street, the bay, the trail, the library, the school, the workshop, or the conversation.

---

## 18. Decision rule

Use this rule before approving any feature:

If it steals attention, cut it.

If it needs child data, redesign it.

If it encourages compulsive use, reject it.

If it cannot be explained to a parent in one sentence, simplify it.

If it does not point back to Frankston, remove it from the public-facing experience.

---

## 19. Recommended first implementation

Use the charter to review one small pilot before launch.

Best first test:

Living Mural Lesson Wall or Scan Less / Notice More Walk.

Why:

- no-login QR page
- printable classroom/family prompt
- clear offline action
- no child data
- measurable through anonymous aggregate visits and partner feedback
- easy for Council, schools, and libraries to understand

## 21. Diagram spec - Screen as Doorway, Not Destination

Use this as the core visual diagram for the charter, website, council checklist, and pilot approval form.

Flow:

QR / Link → Short Prompt → Offline Action → Optional Reflection → Finish.

Design notes:

- The flow must end at "Finish", not at another feed, gallery, or sign-up step.
- The offline action should be visible in the middle of the diagram, not treated as a footnote.
- The diagram should make clear that the screen opens the door and then gets out of the way.
- Frankston/place should be represented as the destination of attention, not the digital interface.

---

## 22. Proof checklist before public pilot

Before any Healthy Technology claim is made in public material, the pilot must show practical evidence of:

- load speed
- no-login test
- no-child-data test
- printable fallback
- mobile readability
- accessibility pass
- visible privacy note
- clear exit prompt
- adult-led group mode

This proof checklist prevents the charter from becoming a values statement without implementation discipline.
