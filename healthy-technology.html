01 Purpose Statement

Healthy technology helps Frankston become more creative, not more screen-bound.

Frankston 2035 - Art That Lives treats technology as a support layer for real places, real people, and real creativity. The screen is not the artwork. The ecosystem is the artwork.

Healthy technology is short, clear, useful, privacy-first, accessible, and connected to offline creativity. It does not compete with Frankston for attention. It points attention back to Frankston.

Parent test

This tool helps my child notice or make something in Frankston, then it clearly ends.

02 Main Document Draft

Healthy Technology Charter: Technology Should Give Attention Back

Definition

Healthy technology is short, clear, useful, privacy-first, accessible, and connected to offline creativity.

Scope

This applies to QR-linked public art pages, classroom tools, library resources, family prompts, poster interactions, walking trails, and sister-city exchanges.

Create, then leave

The preferred journey is simple: open the prompt, understand the task, make or notice offline, finish.

Clear endings

Every experience should make it obvious when the user is done: close this and draw, walk, print, talk, or return to place.

StandardRequirementReason
No infinite scrollPages must have clear boundaries and endings.Prevents passive consumption.
No dark patternsNo manipulative consent, sign-up, navigation, or exit flows.Protects public trust.

No child personal dataDo not ask children for names, emails, photos, profiles, locations, or identifiers.Reduces legal, ethical, and duty-of-care risk.

03 1-Page Summary

The screen opens the door. The work happens in Frankston.

Healthy technology should help people engage with public art, classrooms, libraries, homes, and streets without becoming the main event.

Council can support digital tools when they are low-risk, privacy-conscious, accessible, no-login wherever possible, and linked to clear public value.

Scan.Notice.Make.Finish.

04 Bullet Point Version

Fast rules for any pilot, QR layer, classroom prompt, or public tool.

Hard no

  • No infinite scroll.
  • No dark patterns.
  • No gambling mechanics.
  • No compulsory accounts for public participation.
  • No personal data collection from children.
  • No hidden tracking.

Required behaviours

  • No-login by default.
  • Short sessions.
  • Clear exits.
  • Offline creative prompts.
  • Family mode.
  • Classroom mode.
  • Print mode.
  • Plain English.

Accessibility basics

  • Readable contrast.
  • Mobile readability.
  • Keyboard-friendly interactions.
  • Alt text for meaningful images.
  • Captions or transcripts for audio/video.
  • No meaning carried by colour alone.

Offline action

  • Draw this.
  • Walk here.
  • Notice this.
  • Talk about this.
  • Print this.
  • Bring this to class or the library.

05 Council-Safe Version

A practical support position for councils, schools, libraries, and parents.

Digital layers should be lightweight, accessible, privacy-conscious, and tied to a clear public benefit. The strongest version of the program should still work as a print resource, a guided walk, a classroom sheet, or a library table.

Any public pilot should document risks, permissions, child-safety considerations, accessibility, and data handling before launch.

09 Next Actions Checklist

Use this before any QR, classroom, library, or child-facing pilot goes public.

  1. Name the one real-world action the page supports.
  2. Remove any feature that keeps people scrolling.
  3. Use no-login access by default.
  4. Avoid child personal data entirely.
  5. Add a clear ending and offline next step.
  6. Check accessibility basics.
  7. Provide a printable version.
  8. Measure only what helps improve the pilot.
